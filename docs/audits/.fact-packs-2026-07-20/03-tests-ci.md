## Summary

CI on this repo is decorative, not enforcing. The `Lint` job in `.github/workflows/ci.yml` (pre-commit: formatting/lint/secrets) has failed on **every push to `main` for over a month** (last success 2026-06-19; every main-branch run since, through 2026-07-17, is `failure`), and the 50 most recent workflow runs overall (any branch) are 48 failures / 2 cancelled, 0 successes. There is no branch protection on `main` (404 on the protection API) and most events are direct `push`, so the red CI has never blocked anything. The bash test suite (`test/run-tests.sh`, 5 files, ~68 assertions) is **never invoked by CI at all** — it only runs if a human runs it manually, and `docs/TESTING.md` documents a fictional CI job (`🧪 Test Dotfiles` on `macos-latest`) that does not exist in `.github/workflows/`. Running the suite live today, one of the five test files (`test/test-tmux-dev.sh`, 320 lines) fails 12/14 checks because it tests `scripts/tmux-dev.sh` and `scripts/tmux-session-switch.sh`, neither of which exists anywhere in the repo — and because `test/run-tests.sh` uses `set -e`, that failure silently aborts the runner before the remaining two test files (`test-skill-suite-lint.sh`, `test-workflow-guard.sh`) ever execute. Separately, the current **uncommitted working tree** has already deleted `dotfiles/.claude/skills/` (migrated to `dotfiles/.config/agents/skills/`), which — once committed — will break two of `.pre-commit-config.yaml`'s local hooks that still hardcode the old `dotfiles/.claude/skills/...` path (confirmed locally: exit 127, "no such file or directory"); CI logs from before this migration show those same two hooks passing. The `.secrets.baseline` used by `detect-secrets` is 18 months stale (generated 2025-01-06, zero baselined entries) so that hook fails every run against real, un-triaged findings (nvim `lazy-lock.json`, `openwiki/.last-update.json`, a doc file). The OpenWiki doc-refresh workflow's scheduled cron is deliberately disabled in favor of a local launchd job (`~/Library/LaunchAgents/com.alexwelch.openwiki.plist`, loaded, firing daily at 3am) — but that job has failed its actual doc-generation step for at least 3 straight days (2026-07-18 to 2026-07-20) due to a Node.js native-module ABI mismatch (`better-sqlite3` compiled for `NODE_MODULE_VERSION 127`, Node now needs `147`), and even its fallback push-to-branch step for a second repo (`~/projects/agents/iris`) errors out daily trying to open a PR with no new commits. Only a `commit-msg` git hook is actually installed (`core.hooksPath=.githooks`, one file: normalizes commit messages) — nothing enforces pre-commit or tests at commit time.

## Open questions

- Was the intent for `.github/workflows/ci.yml`'s "Lint" job to actually gate merges, or is it accepted as advisory-only given no branch protection exists? If advisory, is a month of continuous red worth the noise, or should the auto-fixing hooks (trailing-whitespace, markdownlint --fix, shfmt -w, yamlfmt, end-of-file-fixer) be changed to a check-only mode (or auto-commit their fixes) so the job can actually go green?
- Is the `dotfiles/.claude/skills` → `dotfiles/.config/agents/skills` migration (currently uncommitted, per `git status`) intended to land soon? If so, `.pre-commit-config.yaml:35,40` needs its `entry:` paths updated in the same commit or the two skill-lint hooks will start failing (127) the moment CI next runs against a commit that includes the deletion.
- Should `test/test-tmux-dev.sh` be deleted (feature it tests, `scripts/tmux-dev.sh`/`scripts/tmux-session-switch.sh`, doesn't exist) or is a rewrite against the current `hdev.sh`/`hlog.sh` (herdr) workflow intended? Either way, `test/run-tests.sh`'s `set -e` behavior means one broken file currently masks whether 3 other test files even run in a single invocation — worth deciding if that's desired ("fail fast") or accidental (should aggregate and report all 5 files' results).
- Is `docs/TESTING.md`'s CI section meant to describe a workflow that should exist (i.e., wire `test/run-tests.sh` into `.github/workflows/ci.yml`), or is it leftover from an earlier design that was superseded by the current 5-job `ci.yml` and just never updated?
- Is the `better-sqlite3`/Node ABI mismatch breaking the openwiki launchd job a known, accepted-for-now issue, or has it gone unnoticed because the script downgrades failures to non-fatal `WARN`s and the log only surfaces via `/tmp/openwiki-scheduled.log`/`.err` (not any active notification)?
- Is the recurring `openwiki/update` branch on `~/projects/agents/iris` in a state (already-merged, stale local branch) that's causing the daily `gh pr create` failure, and if so should the scheduled script detect "no new commits vs existing open PR" and skip cleanly instead of erroring every day?
- Is `.secrets.baseline` (untouched since Jan 2025) intentionally abandoned in favor of always fixing/ignoring detect-secrets findings ad hoc, or should someone run `detect-secrets scan --baseline .secrets.baseline` and triage the current findings?
- Is the mixed action-pinning strategy (SHA-pinned in `openwiki-update.yml`, floating tags in `ci.yml`) an intentional risk tradeoff (ci.yml doesn't have write permissions/secrets, openwiki-update.yml does) or an oversight?
