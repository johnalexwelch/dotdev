## Follow-ups

- **NEW-04 (low) — `ls`-style command output gets stripped by the tool harness for subagents.** During Task 3 dispatch, the subagent's first `ls -la` call returned only a bare filename; the long-format columns (permissions, owner, timestamp, size) were stripped somewhere in the tool-result pipeline. Workaround: invoking via the absolute path `/bin/ls`. **For Phase 2 production port:** either (a) teach the scope-verification subagent to re-run `git status --porcelain` / `ls -la` from the orchestrator rather than trusting subagent-quoted listings, or (b) note in `## Tuning notes` that subagent prompts should prefer absolute-path invocations for commands whose output is load-bearing evidence. Option (a) is stronger; option (b) is cheaper and partially mitigating. Worth bundling into Phase 2 Task 6 (scope-isolation mechanic).

- **NEW-05 (medium) — Outcome-file path collision when multiple plans both have a Phase N.** The scaffolded skill writes to `docs/executions/.phase-runs/<date>-phase-<N>.md`. In this session, both `2026-04-21-skills-updates-design.md` Phase 1 AND `/tmp/pilot-plan.md` Phase 1 want that exact filename. Resolved ad-hoc by routing the pilot to `2026-04-21-pilot-phase-1.md`, but this is a naming gap. **For Phase 2 production port:** add a `plan_slug` component to the outcome filename — e.g. derive from the plan's filename stem or an explicit `<!-- plan-slug: -->` marker in the plan header. Suggest schema: `<date>-<plan-slug>-phase-<N>.md`, falling back to `<date>-phase-<N>.md` when only one plan is in flight in the repo. Fold into Phase 2 Task 4 (preflight + plan-parsing) or Task 10 (outcome file schema).
