## §0 For Claude Code — read this first

Start at §5.0 Phase 0. Work phases in order — each phase is a vertical slice that ships by itself and can be rolled back independently. Do not skip the pilot (§5.1); it exists specifically to prove FIND-01's fix before anything else depends on `/repo-audit` producing reliable fact-packs.

**Tag semantics.** Every §5 task carries exactly one tag:

- `[auto]` — Claude Code executes without confirmation. File edits, scoped refactors, new-file creation, test/lint runs, git commits.
- `[human]` — Requires user judgment before execution. Config decisions (which settings to copy vs. exclude), scope changes, anything involving a real target repo for dogfood, and any action that would modify a shared/production system.

Mixed tasks are split — no task is half-auto, half-human.

**Audit cross-reference.** Every phase's "Addresses" line cites the stable `FIND-NN` IDs from `docs/audits/2026-04-21-repo-audit.md`. `/execute-phase` commit messages must echo those IDs (`phase-<N>: <Goal> (addresses FIND-NN, FIND-NN)`) for `/post-mortem` and `/describe-pr` to parse.

**Execution-file conventions.** Phase-run outcome files land at `docs/executions/.phase-runs/2026-04-21-phase-<N>.md` (extends the convention already established by the 2026-04-21 skills-updates plan). Pre-phase backup tarballs go to `~/.claude/skills.pre-phase-<N>.tgz`.

**Sync-gate rule.** Main stays clean between phases. Do not start Phase N+1 until Phase N's commit lands on main and its verification recorded green in its outcome file.

## §1 Purpose

This plan turns `docs/audits/2026-04-21-repo-audit.md` into an executable six-phase slice that closes the audit's Top Three (FIND-01 subagent architecture, FIND-02 zero end-to-end dogfood, FIND-03 zero CI/commit-schema gate) plus the smaller medium/low findings within reach in a single plan. The audit's Top Three are a compound risk cluster — fixing them in a single plan is the narrow, achievable outcome; larger items like the brief-mode rollout (FIND-05) and installed-skill regularization (FIND-09) are explicitly deferred to follow-on plans.

This plan is Phase 5 dogfood for the 2026-04-21 skills-updates plan: the point is to exercise `/design-plan` on a real audit with real stakes, surface gaps in the core loop, and feed them into the next audit cycle.

## §2 Problem

The repo-audit of 2026-04-21 produced 15 stable findings (`FIND-01` through `FIND-15`) with a sharp risk cluster at the top. FIND-01 is the closest thing in the audit to an "it doesn't work" finding: `/repo-audit`'s specified fan-out to 13 Explore subagents cannot reliably Write fact-pack files to `docs/audits/.fact-packs-*/` because Explore-type subagents don't have Write-tool access at those paths. During the very audit run that produced this document, 7 of 13 fact-packs required orchestrator-side recovery — meaning the skill's documented map-reduce workflow is emergent, not stable. Until the architecture is fixed (or the fallback is formalized), every audit we run is partly hand-assembled.

FIND-02 compounds it: the entire 4-skill core loop has never executed end-to-end on a real repo. Phase 5 of the prior plan halted at target-repo selection, so every downstream claim — `/execute-phase` commit schema, `/describe-pr` cross-reference parsing, `/post-mortem` FIND-NN tracking, `/setup-worktree` env-file copy, `dry_run=true` semantics, cluster grouping — is specification-only. The three load-bearing contracts of the loop (strict-YAML SKILL.md frontmatter, `phase-<N>: <Goal> (addresses <IDs>)` commit messages, `docs/{audits,plans,executions}/` directory conventions) have no automated gate (FIND-03): a malformed YAML or mis-shaped commit goes undetected until a downstream skill parses it wrong and produces silent garbage.

Surrounding these three: no README or command index (FIND-07) so discovery is per-file across 11 skills, a latent `.claude/settings.local.json` leak in `/setup-worktree` (FIND-06), organizational-context strings in `slack-update` examples (FIND-11), an empty `new-project/` directory of unclear intent (FIND-12), unbounded phase-run retention (FIND-08), non-atomic `.omc/state/` writes (FIND-15), and no failure-notification path for unattended halts (FIND-04).

## §3 Goals and non-goals

**Goals**

- **G1.** Close out FIND-01 with a single-skill edit that either (a) switches `/repo-audit` fan-out to `general-purpose` subagents with explicit Write scope, or (b) formalizes the orchestrator-fallback path that actually worked. Verification: re-run `/repo-audit` on a trivial target and observe ≥12 of 13 fact-packs written by subagents.
- **G2.** Close out FIND-03 with a `.pre-commit-config.yaml` (or equivalent Makefile target) that fails on malformed SKILL.md YAML and on commit messages that don't match `phase-<N>: <Goal> (addresses <IDs>)`. Verification: intentionally malformed YAML is rejected; off-schema commit is rejected.
- **G3.** Scaffold onboarding surface (FIND-07): root `README.md` (purpose + loop diagram + skill matrix + prerequisites), `SKILL_INDEX.md` (table of all 11 skills), and `CLAUDE.md` (house conventions). Verification: a cold reader can identify which skill to run for a given task in <5 min.
- **G4.** Close out FIND-02 with one live dogfood of the loop on a small real target repo. Verification: `docs/executions/.phase-runs/2026-04-21-phase-4-dogfood.md` exists with non-empty outcome sections for all deferred checks (dry-run path, `/setup-worktree` live run, `/describe-pr` live PR, `/post-mortem` live run, cluster-grouping heuristic).
- **G5.** Sweep the medium/low cleanup set (FIND-06, FIND-08, FIND-11, FIND-12, FIND-15) down to either closed or explicitly-deferred-with-owner.
- **G6.** Preserve existing skill voice and do not regress the strict-YAML frontmatter on any of the 6 core-loop SKILL.md files. Manual YAML parse + commit-schema checks must stay green at every phase boundary.

**Non-goals**

- **NG1.** Brief-mode rollout (FIND-05). The 2026-04-22 brief-mode plan is already written; this plan does not execute it. Reason: FIND-05 blocks on Phase 4 dogfood landing first, and stuffing two plans into one violates the "5–6 phases, not a year-long roadmap" constraint.
- **NG2.** Installed-skill frontmatter regularization (FIND-09, FIND-10, FIND-14). Touching `ci-deploy-fix`, `slack-update`, `td-task-management`, `write-to-obsidian`, and `omc-reference` is a 2–3 hour block that belongs in its own plan. Partially addressed: `slack-update` org-context strings (FIND-11) get anonymized as part of Phase 5 because they're a security concern, not a frontmatter concern.
- **NG3.** GitHub Actions CI (FIND-10 adjacent). The repo has no remote and pre-commit hooks are sufficient for the gate FIND-03 demands. Adding Actions would require a remote-push decision (FIND-11 says don't push public without cleanup first).
- **NG4.** Failure-notification primitive (FIND-04). Design-complete but deferred to a follow-on — needs webhook/chat decisions that are out of scope. We will add a lightweight local file primitive (`.omc/state/unresolved-halts.txt`) in Phase 5 if cheap, otherwise defer entirely.
- **NG5.** Anything touching `~/wt/` target worktrees or remote git state. All work lands in `~/.claude/skills/` directly on `main`.

### §5.2 Phase 2 — Pre-commit gate for SKILL.md YAML + commit-message schema (FIND-03)

10. [auto] Commit: `phase-2: add pre-commit yaml and commit-msg gates (addresses FIND-03)`.

## §7 Architecture rules

- **No file larger than its parent skill.** A SKILL.md edit should not balloon past the 2026-04-21 baseline line count by more than 20% in a single phase. If it does, split the edit.
- **No new top-level directories outside `docs/`, `scripts/`, and existing skill directories.** Phase 2 adds `scripts/`; that's the only permitted new top-level directory in this plan.
- **Strict-YAML frontmatter invariant.** Every SKILL.md must parse via `yaml.safe_load` on the frame between the two leading `---` lines. Phase 2's hook enforces this; Phase 1's edit must not break it.
- **Commit-message schema invariant.** Every commit created by `/execute-phase` (or by hand during this plan) must match `^phase-\d+: .+ \(addresses (FIND|NEW|REQ)-\d+(, (FIND|NEW|REQ)-\d+)*\)$` *except* for: Phase 0 preflight commit (none — phase writes no source), the Phase 6 `post-mortem` commit (pure meta, schema-compliant but no FIND cited — allowed), and any baseline/docs/chore commit predating this plan.
- **No circular reads across the 6 core-loop SKILL.md files.** `design-plan` reads `repo-audit`'s output; `execute-phase` reads `design-plan`'s output; etc. The DAG is already clean per audit fact-pack 02; phases here must not introduce a back-edge.
- **No secrets in any new file.** Phase 2's scripts must not read or print file contents beyond what's necessary. Phase 3's docs must not mention user paths beyond `~/.claude/skills/` and `~/wt/`.

## §11 Sync-gate mechanics

Solo dev, no remote, two-commit baseline → lighter sync gate than the default /design-plan recommends.

- **Branch naming.** This plan runs directly on `main` (solo convention). No `refactor/phase-<N>-<slug>` branches — those are for the *target* repo in Phase 4 dogfood, not this repo. If any phase feels risky enough to want isolation, use `/setup-worktree` to create a phase worktree (which will be at `~/wt/skills/phase-<N>/`), but default is direct-to-main.
- **PR policy.** None — no remote, no reviewers, solo loop.
- **Commit schema.** Every commit from Phase 1 onward matches `phase-<N>: <Goal> (addresses FIND-NN[, FIND-NN]*)`. Phase 2's hook enforces this starting with its own commit.
- **Main-clean-before-next-phase.** After each phase's final commit: `git status` clean, `make check` green, phase-run outcome file exists. Only then start the next phase. `/execute-phase auto_proceed=true` is fine for Phase 1 → Phase 2 and Phase 2 → Phase 3 (low-risk), but should be `auto_proceed=false` at the Phase 3 → Phase 4 boundary (Phase 4 has a human gate and touches an external repo).
- **Mid-phase failure.** If any phase's verification fails: halt, write the failure to the outcome file, run the phase's Rollback, and either (a) revise the phase in-place and retry once, or (b) revise this plan via `/design-plan mode=revise`. Do not "just fix it ad-hoc and move on" — that's exactly the pattern that produced FIND-01's emergent-not-documented fallback.
- **Backup cadence.** `~/.claude/skills.pre-phase-<N>.tgz` before each phase. Delete old tarballs once the phase lands and verification sticks (keep N-1 as insurance).
